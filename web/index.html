01 · Por quê

Sem esteira, cada publicação é um risco.

Sem automação, problemas de modelagem, de relatório e de reuso só aparecem tarde, muitas vezes em produção. Esta esteira move a qualidade para a esquerda, direto no Pull Request.

Sem padronização

Cada dev publica do seu jeito; boas práticas de modelo e relatório ficam a critério individual.

Erro tardio

Problemas de DAX, colunas mal configuradas e visuais fora do padrão só são pegos tarde demais.

Sem colaboração & reuso

Modelos e datasets duplicados por falta de reaproveitamento: mais conexões no gateway, mais custo e retrabalho.

Deploy sem trava

Nada impede subir um modelo problemático. Falta um gate que bloqueie o que não deve passar.

02 · Arquitetura

Do commit ao Fabric, em duas lanes.

Ao abrir o PR, o GitHub roda a Etapa 1 (estática). Aprovado e mergeado, a integração Git entrega ao Fabric, onde entra a Etapa 2 (dinâmica) até produção.

03 · Modelo de validação

São duas etapas que se completam.

A validação não é um passo só. Ela acontece em dois momentos com objetivos diferentes: antes do merge (barata, sem capacity) e depois no Fabric (com dados reais).

01

Estática · no GitHub

Roda no Pull Request · runners efêmeros · sem Fabric

  • Metadata: integridade dos arquivos
  • Report Rules: boas práticas do relatório
  • BPA: boas práticas do modelo
✔ Implementada
02

Dinâmica · no Fabric

Roda pós-deploy · exige modelo publicado + refresh

  • Refresh test: o modelo atualiza sem erro
  • DAX tests: medidas retornam o valor esperado
  • Performance: tempo de resposta dentro do limite

04 · Inteligência & Colaboração

Valida só o que mudou.

Vários times podem colaborar no mesmo repositório, trabalhando em conjunto. Um job de descoberta faz o diff do PR contra a base e valida apenas os projetos PBIP alterados, então a esteira foca só no que mudou. Inteligência e colaboração, sem travar o time.

05 · Customização & Parametrização

A severidade decide se bloqueia.

Parametrização BPA: o campo CustomSeverity controla como as violações da análise do BPA são classificadas e o que reprova o build (por exemplo: só o nível 3 bloqueia; o resto é aviso ou sugestão). As outras validações não têm esse campo: o Report Rules é parametrizável pelo seu arquivo de regras (mudar limites ou desativar regras) e o Metadata roda checagens fixas de integridade. Foco em customização e parametrização.

06 · Destino

Aprovado? Vai para o Fabric.

A integração Git do Fabric leva a definição do item. Modelos import chegam prontos, mas precisam de um refresh para materializar os dados (início da Etapa 2).

Git Integration

Merge na main → o workspace de Dev recebe Semantic Model e Report automaticamente.

Refresh = dados

Integração ≠ refresh. Um Refresh now (ou via REST API) popula o modelo. Dados inline dispensam gateway.

Dev→ deployment pipeline →Prod(Fase 2)
